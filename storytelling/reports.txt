VARIÁVEIS DE AMBIENTE
A fim de manter a segurança do projeto, é uma boa prática não enviar certas variáveis para o repositório. Para isso:
- .env: pip install python-dotenv
- No venv, criar o arquivo e configurar as variáveis.
- Nos arquivos que utilizam as variáveis de ambiente:
  import os
  from dotenv import load_dotenv
  load_dotenv()
  Onde for usar as variáveis: = str(os.getenv('NOME_VARIÁVEL'))

CRIANDO APPS
Diferentemente do startproject, que gera o setup inicial para o desenvolvimento de todo o projeto, o comando "python manage.py startapp <nome do app>" serve para criar as pequenas aplicações que compõem o projeto.
Em outros termos, enquanto o startproject trata do macro, o startapp lida com o micro.
- Após criar o app, ir em settings.py a adicioná-lo no INSTALLED_APPS

CRIANDO VIEWS
Para as respostas gráficas de cada aplicação, é necessário trabalhar sobre os arquivos views.py de cada um. De maneira geral, têm-se em views.py:
- from django.http import HttpResponse
- def index(resquest):
    return HttpResponse(<html da página>)
Um vez criada a interface, vai-se em urls.py na pasta principal do projeto e executa-se duas atividades principais:
- importação da função responsável pela resposta gráfica;
- listagem do respectivo path

ISOLAMENTO DE URLS
O método acima exige que todos os pths estejam agrupados em uma lista em um arquivo só. Contudo, em termos de gerenciamento, isso é uma estrtégia ruim. Quantos paths não teriam de estar aglutinadas em uma lista? Quão difícil não seria trabalhar com isso? Para evitar esse problema, segue-se:
- Na pasta do app, cria-se outro urls.py, voltado apenas às rotas do prórpio app. Nele:
  from django.urls import path
  from <app>.views import <função de retorno gráfico, ex: index>
  urlpatterns = [path('', index)]
- Retornando ao urls.py do projeto:
  from django.urls import include
  urlpatterns = [
    (...), 
    path('', include('<app>.urls')),
  ]

TEMPLATES
A partir de agora, não utilizaremos a mesma abordagem de quando criamos as views, quando usamos HttpResponse e etc. O desafio é organizarmos os templates html e consumi-los, para isso:
- No diretório base, criar pasta templates com os templates html a serem consumidos;
- Em settings.py (setup), apontar para os templates via INSTALLED_APPS e, em TEMPLATES, 'DIRS': [os.path.join(BASE_DIR, 'templates')]
- Em views.py, alterar index para consumir request e retornar render(request, 'index.html')

ARQUIVOS ESTÁTICOS
Não basta contudo, carregar os templates se os arquivos estáticos, ou seja, aqueles que servem para definir a arte visual da aplicação, não estiverem sendo utilizados. Para isso, segue-se:
- Configuração
  Em settings.py, na parte de configurações estáticas, criar
  1. STATICFILES_DIR = [os.path.join(BASE_DIR, 'setup/static')] - observe, portanto, que deve ser passado uma pasta static em setup; esse diretório apontado é o ponto de consumo de arquivos estáticos da aplicação.
  2. STATIC_ROOT = os.path.join(BASE_DIR, 'static') - esse, por sua vez, refere-se ao diretório para as manipulações que o django executar.
  3. python manage.py collectstatic
- Adaptação
  No index.html, o template, indicar o consumo de arquivos estáticos para o django via {% load static %} no início
  Ainda em index.html, é necessário apontar onde os arquivos estáticos serão utilizados. Para isso, onde for necessário, implementar: "{% static '<caminho/do/arquivo/consumido>' }"
  Bizu: como todos os arquivos precisam ser apontados, selecionar os múltiplos pontos a serem aletrados segurando Alt (VSCode) e implementar.

OUTRAS PÁGINAS
As aplicações precisam integrar suas páginas. Até então, arquitetou-se o index, faltam os desdobramentos das demais funções.
Resumidamente, tudo o que aplicou-se para o index, plica-se agora. Contudo, um vez configurada, a página precisa ser apontada pelas demais. Para isso:
- em urls.py, adicionar parâmetro name=<nome da página>
- no html redirecionador, pôr nos href estrutura semelhante a "{% url '<nome da página>' %}"
  Bizu: Crtl+F (pesquisa) -> clica na seta à esquerda e faz o replace para múltiplas incidências.
